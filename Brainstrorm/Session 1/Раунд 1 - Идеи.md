Участник 1 

**Идея 1**

**Где:** Планирование сроков — разработка ИКГ уровня 2 (Рекомендации, G1). Интегрированный комплексный график с логическими связями CPM-модели.

**Что:** Заменить детерминированный CPM-график вероятностной PERT-моделью с бета-распределениями длительностей работ и запустить Монте-Карло (раздел 4 статьи) для генерации распределения сроков завершения проекта. На выходе — не одна дата, а кривая вероятностей: P50, P80, P90 для каждого ключевого события. Критерии Вальда/Гурвица (раздел 3) применяются для выбора целевой даты в зависимости от рискового аппетита заказчика.

**Зачем:** Прямая замена необоснованным детерминированным графикам, которые «достаточно быстро теряют актуальность» и «изменяются по 5–8 раз» (G1). Даёт обоснованные резервы вместо политически мотивированных дат. Снижает риск каскадных организационных коллизий на уровнях 3–4.

**Почему это сработает:** Модель формализует именно ту задачу, которую ставит G1: CPM-модель + количественный анализ рисков сроков. PERT + Монте-Карло — стандартный метод QRA (H2), но статья добавляет к нему критерии принятия решений и итеративную оптимизацию, чего в стандартных инструментах нет.

**Идея 2**

**Где:** Количественный анализ рисков графика и стоимости — QRA (Рекомендации, H2). Определение реалистичных сроков и обоснованных резервов.

**Что:** Применить полный цикл модели: PERT → Монте-Карло → выявление критических и субкритических путей (раздел 7 статьи) → локальная оптимизация сегментов критического пути (раздел 6) → пересчёт. Итеративно переходить от критического пути к следующему, оптимизируя каждый до тех пор, пока не найден условно-оптимальный план P_opt.

**Зачем:** Текущая практика QRA даёт только диагностику (распределение сроков), но не прескриптивный результат — какие именно работы ускорить и на сколько. Итеративная оптимизация из статьи даёт конкретный оптимизированный план, а не просто статистику.

**Почему это сработает:** Итеративная оптимизация критических путей (раздел 7) — ключевое отличие модели от стандартного QRA. При каждой итерации оптимизируется текущий критический путь, после чего пересчитывается весь граф — субкритический путь может стать критическим. Это математически корректная процедура.

**Идея 3**

**Где:** Тепломонтажные работы (ТМР) — сварка трубопроводов, определяющая критический путь графиков (Construction.md, основной период).

**Что:** Построить GERT-модель (раздел 5) с вероятностными переходами для процесса монтажа и сварки трубопроводов. В узлах ветвления — вероятность прохождения контроля сварных соединений (NDE) с первого раза vs. ремонт vs. повторная сварка. Каждый узел — дерево решений: A_k = (µ_k, C_k, R_k). Оптимизация выбора технологии сварки, последовательности монтажа и стратегии контроля.

**Зачем:** ТМР лежат на критическом пути. Каждый дефект сварки — петля переделки. GERT моделирует эти петли, а оптимизация минимизирует суммарное время с учётом вероятности брака. Потенциал сокращения критического пути на недели-месяцы.

**Почему это сработает:** GERT-сети с обратными дугами (петлями) были спроектированы именно для моделирования процессов с повторными циклами. Сварка трубопроводов — классический такой процесс: сварка → контроль → годен/ремонт/перерезка. Статистика по дефектности сварных соединений накоплена.

**Идея 4**

**Где:** Стратегия инвестирования в подготовительный период (Рекомендации, С2). Фундаментальный выбор: больше вложить в подготовку (снизить строительные риски) vs. минимизировать ранние вложения (сохранить гибкость выхода).

**Что:** Формализовать задачу как задачу принятия решений в условиях неопределённости (раздел 3). Альтернативные траектории A = {Γ1, …, Γn} — различные стратегии распределения инвестиций. Состояния природы S = {s1, …, s_m} — сценарии (проект продолжается, проект отменяется, задержка лицензирования и т.д.). Utility u(Γ_i, s_j) — NPV или полная стоимость владения. Применить критерий Гурвица с параметром α, зависящим от бизнес-модели (госфинансирование → низкий α, экспорт с внешним финансированием → высокий α).

**Зачем:** C2 описывает эту дилемму качественно и приводит примеры (Wylfa, VC Summer, Sizewell C, Балтийская АЭС). Модель даёт количественную основу для принятия решения: при каком уровне инвестиций в подготовку математическое ожидание потерь минимально.

**Почему это сработает:** Задача уже сформулирована в С2 как выбор между стратегиями при неопределённости — модель Дмитришина формализует именно этот класс задач. Критерий Гурвица позволяет настроить баланс оптимизма/пессимизма под конкретный проект.

**Идея 5**

**Где:** Контрактация оборудования длительного цикла изготовления — ОДЦИ (Рекомендации, J1; Приложение 1, раздел 9). Шесть волн поставок оборудования для двух блоков (Construction.md).

**Что:** Построить стохастическую модель графика поставок: каждая единица оборудования — дуга с вероятностным временем изготовления и поставки. Монте-Карло для оценки вероятности задержки поставки относительно потребности монтажа. ML-модель (XGBoost/CatBoost, раздел 9) обучается на данных о фактических задержках поставщиков для прогнозирования T_ij. Оптимизация сроков размещения заказов по критерию just-in-time (Приложение 1, 9.4).

**Зачем:** Несвоевременная поставка ОДЦИ — один из ключевых рисков. Слишком ранний заказ — замораживание средств и затраты на хранение. Слишком поздний — простой монтажа. Модель оптимизирует момент заказа с учётом неопределённости.

**Почему это сработает:** ML на исторических данных поставщиков (задержки, качество, объёмы) повышает точность прогноза длительности изготовления по сравнению с экспертными оценками. Монте-Карло показывает буфер, необходимый для заданной надёжности поставки.

**Идея 6**

**Где:** Пусконаладочные работы, этап А — подэтапы А-0 … А-4 (Construction.md). Планирование методом «от обратного» через этапы ПНР (Рекомендации, G6).

**Что:** Построить GERT-модель этапов ПНР с обратным ходом от целевой даты физпуска. Вероятностные переходы на каждом подэтапе: А-1 (испытания оборудования) → успех/доработка; А-3 (ХГО) → успех/повтор холодной фазы. Критерий Вальда (максимин) для определения безопасных сроков начала каждого подэтапа — поскольку ПНР связаны с ядерной безопасностью.

**Зачем:** Этапы ПНР последовательны и критичны — задержка на любом подэтапе сдвигает пуск. Вероятностная модель даёт обоснованные резервы времени для каждого подэтапа. Критерий Вальда обеспечивает консервативный подход, адекватный ядерной области.

**Почему это сработает:** Таблица критериев в статье (раздел 3) прямо указывает: критерий Вальда применяется для «safety, critical operations, regulatory requirements» — это точно описывает ПНР.

**Идея 7**

**Где:** Мониторинг финансовых рисков проекта (Рекомендации, С3). Оценка стоимости от класса 5 до класса 3 (G2).

**Что:** Многокритериальная оптимизация (раздел 10): вектор F(Γ) = {T(Γ), C(Γ), R(Γ), Q(Γ)} → min. Построить Парето-фронт для каждого варианта графика: найти множество решений, где нельзя сократить срок без удорожания, и наоборот. Визуализировать фронт для заказчика. При итеративном уточнении стоимости (от класса 5 к классу 3) перестраивать Парето-фронт, сужая неопределённость.

**Зачем:** Заказчик получает не одну точку «срок X, стоимость Y», а карту компромиссов. Позволяет обоснованно выбирать баланс сроки/стоимость/риск для директивного графика уровня 1.

**Почему это сработает:** Многокритериальная оптимизация на графах с атрибутами (раздел 10) — финальный компонент модели, спроектированный для задач именно такого класса: множество целей, множество ограничений, необходимость выбора.

**Идея 8**

**Где:** Управление изменениями проекта — change control (Рекомендации, D3). Оценка влияния изменений на сроки и стоимость.

**Что:** При каждом проектном изменении (design change) запускать пересчёт модели: обновить длительности затронутых дуг → Монте-Карло → новое распределение критического пути → оценка влияния на P80. ML-модель (раздел 9) классифицирует изменение по степени риска на основе обученных паттернов (какие типы изменений исторически вызывали наибольшие задержки).

**Зачем:** D3 описывает проблему: изменения после заморозки проекта — ключевой фактор задержек. Модель даёт мгновенную количественную оценку последствий каждого изменения, что позволяет обоснованно принимать/отклонять запросы на изменения.

**Почему это сработает:** ML-классификация рисков изменений обучается на данных прошлых проектов, где последствия изменений уже известны. Монте-Карло пересчитывается быстро при изменении параметров нескольких дуг.

**Где:** Строительство ВЗиС — временных зданий и сооружений (Приложение 1, разделы 13, 25). Пионерная база, СМБ, бытовые городки, дороги, энергоснабжение.

Целевая функция: минимизация времени до готовности критических объектов СМБ (БРХ, арматурное хозяйство, энергоснабжение) с учётом ограничений по ресурсам (бригады, техника). Динамическое программирование для оптимальной расстановки приоритетов.

**Почему это сработает:** Задача сводится к оптимизации сегмента графика с ограничениями — это прямо описано в разделе 6 статьи (линейное программирование, динамическое программирование, стохастическая оптимизация).

**Идея 10**

**Где:** Серийность и стандартизация — строительство второго и последующих блоков (Рекомендации, O1). Шаг ввода блоков: 12, 18 или 24 месяца (G1).

**Что:** ML-компонент модели (раздел 9) обучается на данных фактического выполнения работ 1-го блока. Кластеризация работ (K-means, DBSCAN) выявляет типовые паттерны задержек. Для 2-го блока модель корректирует вероятностные оценки длительностей, используя «кривую обучения» (learning curve). Детектор аномалий (Isolation Forest, автоэнкодер) сигнализирует, если фактический ход работ на 2-м блоке отклоняется от модели.

**Зачем:** Серийность — ключевой фактор снижения стоимости (O1). Но эффект обучения нужно формализовать и отслеживать. ML позволяет прогнозировать реалистичное ускорение на следующих блоках и обнаруживать аномалии раньше, чем они станут критичными.

**Почему это сработает:** Между блоками одной площадки — максимальная однородность данных: тот же проект, те же подрядчики, та же площадка. ML-модели на таких данных дают высокую предсказательную силу. Кластеризация и детекция аномалий (раздел 9) — стандартные задачи для этих методов.

**Идея 11**

**Где:** Выбор модели контрактации проекта (Рекомендации, I1) и схемы ценообразования (I2). Распределение рисков в контракте (I4).

**Что:** Формализовать выбор контрактной модели (EPC-turnkey, split-package, alliance) как задачу принятия решений (раздел 3). Альтернативы Γ — контрактные модели. Состояния природы S — сценарии реализации (задержки лицензирования, инфляция, валютный курс, дефицит рабочей силы). Для каждой комбинации (Γ_i, s_j) — оценка общей стоимости с учётом контрактного распределения рисков. Критерий Сэвиджа (минимакс сожалений) — для минимизации потерь от неоптимального выбора.

**Почему это сработает:** Критерий Сэвиджа (раздел 3) предназначен для ситуаций с ограниченной информацией и множеством альтернатив — это точно описание ситуации выбора контрактной модели на стадии подготовки.

**Идея 12**

**Где:** Планирование сдачи помещений — график передачи помещений из строительства в монтаж (Рекомендации, G10). Координация строительных, электромонтажных, ТМР на одном объекте.

**Что:** Многомерный предиктивный граф (раздел 8): каждое состояние x_i = (t_i, a_i) включает время и вектор атрибутов (% готовности строительства, монтажа, электрики, вентиляции помещения). Модель GNN (Graph Attention Network, раздел 9) обучается на данных о фактическом порядке сдачи помещений на референтных проектах. Прогнозирует реалистичные сроки готовности каждого помещения с учётом зависимостей между смежными помещениями.

**Зачем:** G10 описывает, что график передачи помещений — координирующий документ между строителями и монтажниками. Ошибки планирования здесь приводят к простоям монтажных бригад или к работам «поверх» незавершённого строительства. GNN учитывает пространственные и технологические связи между помещениями.

**Почему это сработает:** Графовые нейронные сети (раздел 9) проектировались для данных с топологией — здания АЭС имеют чёткую пространственную и технологическую структуру зависимостей между помещениями.

**Идея 13**

**Где:** Холодно-горячая обкатка реакторной установки — подэтап А-3 (Construction.md). Фазы А-3.1 (холодная) и А-3.2 (горячая).

**Что:** GERT-модель с вероятностными переходами: гидравлические испытания → годен/не годен → ремонт/повтор; циркуляционная промывка → норма/повторная промывка; горячая обкатка → параметры в норме/доработка. **Почему это сработает:** Автоэнкодер на нормальных данных обкатки (из референтных проектов) хорошо детектирует аномалии в потоковых данных — это стандартная задача предиктивной аналитики в промышленности.

**Идея 15**

**Где:** Адаптация проекта под площадку и национальные требования (Рекомендации, B2). Прелицензионные консультации с регулятором.

**Что:** ML-классификатор (раздел 9, XGBoost/CatBoost) обучается на данных прошлых проектов: тип площадки, страна, требования регулятора → длительность лицензирования, количество итераций замечаний, задержки. **Где:** подготовительный период, лицензирование до первого бетона — _Construction.md_ (этапы 1–2), **B3/B4/B5** в «Рекомендациях»  
**Что:** применить модель для вероятностного маршрута лицензирования: построить сеть разрешений, экспертиз и пакетов документации, прогнать Монте-Карло по срокам рассмотрения, а критериями выбора сценария определить, какие пакеты подавать раньше, какие дробить, а какие объединять.  
**Зачем:** сократить риск срыва даты первого бетона, уменьшить простой проектных и строительных ресурсов, снизить стоимость ожидания.  
**Почему это сработает:** лицензирование — жёсткий «воротный» процесс с ветвлениями и возвратами на доработку; именно для таких неопределённых сетевых процессов интегрированная модель особенно уместна.

**Где:** **C2/C3** «Рекомендаций» — стратегия инвестирования в подготовительный и основной периоды, непрерывный мониторинг финансовых рисков  
**Что:** использовать модель для выбора оптимального объёма ранних вложений: сколько инвестировать в проектирование, площадку, ранние закупки и подготовительную инфраструктуру до FID, а сколько оставить на основной период.  
**Зачем:** минимизировать суммарную приведённую стоимость проекта и одновременно ограничить объём средств под риском потери при остановке.  
**Почему это сработает:** здесь есть прямой конфликт целей — больше подготовки снижает строительные риски, но увеличивает sunk cost; это классическая задача многокритериальной оптимизации под неопределённостью.

**Где:** **J1** «Рекомендаций» и организационно-технологический порядок опережающей контрактации в Приложении 1  
**Что:** применить модель для оптимизации перечня и очередности ранних закупок оборудования длительного цикла: какие позиции брать в опережающую контрактацию, в какой волне, у каких изготовителей и при каком уровне готовности ИДЗ/ТУ.  
**Зачем:** сократить риск дефицита оборудования на критическом пути, уменьшить задержки выпуска РД и монтажа, снизить стоимость срочных решений.  
**Почему это сработает:** длительные циклы изготовления хорошо ложатся в PERT/GERT-структуру, а неопределённость по поставщикам, разрешениям и качеству — в вероятностное моделирование и выбор по критериям риска.

**Где:** **K1/K2** «Рекомендаций», перечень работ подготовительного периода и критерии готовности площадки в Приложении 1  
**Что:** собрать единую модель готовности площадки: дороги, ж/д, временные здания и сооружения, СМБ, энергоснабжение, водопонижение, укрепление грунтов, охранный периметр, городок строителей. Оптимизировать последовательность и ресурсы этих работ.  
**Зачем:** уменьшить вероятность «формально можно бетонировать, фактически площадка не готова», сократить потери от несогласованности фронтов работ.  
**Почему это сработает:** это не одна операция, а плотный узел взаимозависимых предпосылок; интегрированная модель позволяет видеть, какие именно элементы реально ограничивают переход к первому бетону.

**Где:** синхронизация графиков 3 и 4 уровней и **G13** «Рекомендаций»  
**Что:** применять модель как «детектор рассинхронизации» между верхнеуровневым и детальным календарно-сетевым планированием: прогнозировать, в каких местах график 4 уровня почти гарантированно оторвётся от графика 3 уровня, и заранее перестраивать последовательность работ.  
**Зачем:** повысить управляемость графика, сократить фиктивную «плановую» отчётность и реальные просрочки на критическом пути.  
**Почему это сработает:** когда нижний уровень планирования теряет связь с выполнимым верхним уровнем, проблема становится статистически заметной раньше, чем проявляется в фактическом срыве.

**Где:** **H1** «Рекомендаций» — интегрированная система управления рисками проекта  
**Что:** превратить реестр рисков в формализованный сценарный граф: каждый риск связать с вероятностью, влиянием на сроки/стоимость/качество и набором мер реагирования; затем оптимизировать портфель мер снижения рисков.  
**Зачем:** сократить суммарный риск-экспозёр проекта и убрать «слепые зоны», где риски ведутся разрозненно.  
**Почему это сработает:** сам документ прямо требует непрерывного реестра рисков; интегрированная модель даёт способ не просто хранить реестр, а принимать по нему оптимизационные решения.

**Где:** **M1** и **D2** «Рекомендаций» — PLM/BIM/цифровые двойники и проверка коллизий  
**Что:** объединить 3D/4D-модель станции с календарно-сетевой моделью: искать не только геометрические, но и временные коллизии (наложение фронтов, конфликт доступа, блокирование монтажных зон), а ML использовать для прогноза «точек будущих коллизий».  
**Зачем:** сократить переделки, уменьшить простои монтажа, повысить качество подготовки производства.  
**Почему это сработает:** здесь как раз встречаются все шесть компонентов — сеть работ, стохастика, цифровая геометрия, выбор сценариев, обучение на факте и многокритериальная оценка.

**Где:** **D1** и **D3** «Рекомендаций» — зрелость проекта до первого бетона и управление изменениями  
**Что:** использовать модель для расчёта «порога допустимого первого бетона»: при каком уровне готовности проектной документации, данных поставщиков и зафиксированной конфигурации переход к строительству уже выгоднее, чем дополнительная задержка ради доработки проекта.  
**Зачем:** снизить вероятность каскада проектных изменений после старта строительства и связанного удорожания.  
**Почему это сработает:** это не бинарный вопрос «готов/не готов», а оптимизационная граница под неопределённостью, которую можно считать, а не определять экспертно «на глаз».

**Где:** **D4/D5** «Рекомендаций» и раздел _Construction.md_ о работах на СМБ  
**Что:** применять модель для выбора степени укрупнения и переноса работ со стройплощадки на СМБ/завод: какие узлы собирать укрупнённо, где делать предмонтаж, какой размер блоков оптимален для конкретной логистики и механизации.  
**Зачем:** сократить продолжительность основного периода, снизить трудоёмкость на площадке, улучшить качество сборки.  
**Почему это сработает:** выгода от укрупнения зависит от ограничений транспорта, кранов, производственной базы и графика; это именно задача локальной и глобальной оптимизации с множеством ограничений.

**Где:** основной период сооружения, **ТМР и сварка трубопроводов как критический путь**  
**Что:** построить детальную вероятностную модель тепломонтажных работ: трубопроводы, сварка, НК, устранение дефектов, промывки, передача под ПНР; на её основе перераспределять бригады, оборудование и окна доступа между объектами.  
**Зачем:** удержать главный фактический критический путь проекта, уменьшить риск каскадного сдвига монтажа и ПНР.  
**Почему это сработает:** когда узкое место известно заранее, интегрированная модель позволяет не просто фиксировать отставание, а искать наилучшее компенсирующее решение по времени, стоимости и риску.

**Где:** **D7** «Рекомендаций» и этапы ПНР А–Г в _Construction.md_  
**Что:** планировать проект «обратным ходом» от целевой даты физпуска/энергопуска: от этапов ПНР назад к готовности систем, завершению монтажа, поставкам, FAT/SAT и выпуску документации.  
**Зачем:** сократить риск того, что стройка формально завершает объекты, но блок не готов к вводу.  
**Почему это сработает:** ввод — это не конец линейной цепочки, а набор жёстких условий готовности; обратное планирование особенно хорошо работает в вероятностной сетевой модели.

**Где:** **A4** «Рекомендаций» и объекты схемы выдачи мощности / внешней инфраструктуры в _Construction.md_  
**Что:** расширить модель с уровня «строим блок» до уровня «строим блок + выдачу мощности + внешнюю инфраструктуру»: ЛЭП, подстанции, трансформаторы, КРУЭ, техническое водоснабжение, внешние сети.  
**Зачем:** убрать ложную оптимизацию, когда площадка готова, а внешний контур не позволяет выдать мощность или нормально обеспечить объект ресурсами.  
**Почему это сработает:** внешняя инфраструктура часто живёт в отдельном контуре управления, хотя фактически влияет на общий срок и риск проекта не меньше, чем часть работ внутри ограждения.

**Где:** **J4** «Рекомендаций» и контрактный механизм изменений **I6**  
**Что:** использовать модель для сценариев санкций/экспортного контроля: заранее считать, какие узлы критичны к замене поставщика, где потребуется перепроектирование, какие альтернативы минимизируют regret при внезапном ограничении импорта.  
**Зачем:** уменьшить вероятность многолетних задержек из-за вынужденной замены компонентов и переработки документации.  
**Почему это сработает:** это классическая область для критериев выбора в условиях неполной информации — особенно minimax regret и сценарный анализ.

**Где:** **L3/L4** «Рекомендаций» и готовность городка строителей в Приложении 1  
**Что:** применить модель для помесячного прогноза потребности в персонале по специальностям с увязкой к жилью, транспорту, столовым, доступу на площадку и графику работ; оптимизировать темп мобилизации персонала.  
**Зачем:** избежать как дефицита критических специалистов, так и преждевременного «перенасыщения» площадки людьми и затратами.  
**Почему это сработает:** численность персонала — зависимая величина от сценария графика и фронтов работ; её можно считать как часть общей модели, а не отдельным HR-документом.

**Почему это сработает:** Детерминированные графики создают жесткие, искусственные пики потребности в людях. Алгоритмы локальной оптимизации автоматически «размажут» эти пики, выдав взвешенную потребность в ресурсах без сдвига даты пуска.

**Зачем:** Сокращение этапов «холодно-горячая обкатка» и выхода на МКУ за счет оптимизации последовательности тестов.
**Почему это сработает:** ПНР — стохастический процесс, где каждый дефект тянет за собой цепочку задержек. Модель поможет оптимизировать порядок операций для минимизации совокупного времени ожидания и переделок.

**Идея №8**
**Где:** Приложение 1, Раздел 12 (Графики работ); Рекомендации, Раздел G (Планирование).
**Что:** Внедрение итеративной оптимизации критических путей (раздел 7 статьи) на этапе формирования сводного графика сооружения АЭС для РФ.
**Зачем:** Создание устойчивого к возмущениям графика (robust schedule), который допускает отклонения без срыва ключевых событий.
**Почему это сработает:** Большинство графиков «ломаются» при первом отклонении. Итеративная оптимизация с генерацией множества сценариев позволит заложить в график «буферы» там, где вероятность срыва максимальна.

**Идея №9**
**Где:** Рекомендации, Раздел C3 (Финансовые риски).
**Что:** Сценарный анализ и стресс-тестирование финансовой модели АЭС с использованием вероятностных моделей (раздел 4 статьи) для расчета резервов на непредвиденные расходы.
**Зачем:** Формирование объективного управленческого резерва бюджета, достаточного для покрытия 80-90% рискованных сценариев, без завышения стоимости для инвестора.
**Почему это сработает:** Сейчас резервы часто берутся экспертно или % от сметы. Монте-Карло позволит рассчитать точный размер резерва для заданного уровня уверенности (например, P80).

**Идея №10**

**Зачем:** Исключение ситуаций, когда пионерная база построена, а под неё меняется проект площадки, или когда пропускная способность СМБ ограничивает темпы основного периода.
**Почему это сработает:** Затраты на ВЗиС значительны. Оптимизация здесь позволяет высвободить средства для основного строительства без риска для сроков.
