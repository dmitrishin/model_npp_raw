### Единственный открытый спор: L-01 vs B-02

Участник 6 аргументированно предложил заменить L-01 (offset между блоками) на B-02 (инвестиционная стратегия с моделированием сценариев отмены):

**Аргументы «за» замену (уч. 6):**

- Offset на 90% определяется политикой (МПС, транши экспортных кредитов), а не математикой
- B-02 защищает акционерный капитал от миллиардных убытков при заморозке проекта (кейсы V.C. Summer, Балтийская АЭС)
- Критерии Вальда/Сэвиджа жизненно необходимы для решения о FID

**Аргументы «за» сохранение L-01 (уч. 1, 3, 5, 8):**

- Offset — стратегическое решение с прямым эффектом для серийных проектов (15–30% экономии)
- Модель нужна не для «назначения» offset, а для анализа чувствительности внутри допустимого диапазона
- B-02 — важная, но узкоспециализированная задача предконтрактной фазы

**Решение:** Большинством (6/8) L-01 сохранена в ТОП-10. B-02 рекомендована как приоритет «второй волны».

## УТВЕРЖДЁННЫЙ ФИНАЛЬНЫЙ ТОП-10

На основании позиций всех 8 участников Раунда 6 утверждается следующий список:

### 1. A-01 — Единая GERT-макромодель подготовительного периода

**Консенсус: 8/8.** Абсолютный лидер всего процесса. Подготовительный период (2–8 лет) — главный источник фатальных задержек.

**Финальная конфигурация (по итогам критики):**

- Федеративная архитектура: 3 отдельных подграфа (лицензирование, проектирование, контрактация) + 5–7 перекрёстных триггеров
- Политические решения — как сценарные параметры, не стохастические узлы
- Для экспорта: нечёткая логика вместо фиксированных вероятностей
- Обязательный анализ чувствительности и отсечение слабозначимых ветвей
- MVP: укрупнённый GERT (10–15 ключевых событий) + детальные подмодели для критических петель

### 2. A-03 — Предиктивный критический путь с двухуровневым ранним предупреждением

**Консенсус: 8/8.** Единственная идея, превращающая модель из инструмента планирования в инструмент оперативного управления.

**Финальная конфигурация:**

- Гибридный подход: rule-based + Isolation Forest (тактический уровень), GNN — только для топ-200 критических операций
- Обязательная расшифровка каждого алерта через top-drivers (Explainable AI)
- Доверительный интервал при каждом предупреждении
- Ресурсные конфликты (краны, F-02) — «зашиваются» внутрь как ограничения
- Переход к полной GNN — после 12–18 мес. накопления данных

### 3. A-04 — Непрерывный байесовский Readiness Index

**Консенсус: 8/8.** Объективный количественный шлюз перед Первым бетоном (N1 Рекомендаций).

**Финальная конфигурация:**

- Не скаляр, а **вектор готовности** по 5–6 доменам (лепестковая диаграмма)
- Hard-stop (жёсткие вето): критичный «красный» параметр блокирует переход независимо от общего RI
- Динамические веса (вес лицензирования растёт по мере приближения к ПБ)
- Разделение на «сырые» данные (жёстко из ERP) и экспертные оценки (аудит)
- Интеграция с A-01 для автоматического расчёта

### 4. D-01 — Двухфазная калибровка: кластеры аналогов + байесовское обновление

**Консенсус: 7/8.** Фундамент достоверности всех остальных моделей.

**Финальная конфигурация:**

- Иерархические кластеры (5–7 ключевых признаков вместо 15–20)
- Механизм «дрейфа кластера» — автоматический пересмотр принадлежности проекта к аналогу по мере накопления факта
- Стандартизация WBS across проектов для сопоставимости
- Экспертные веса для корректировки при недостатке статистики
- Доверительный интервал качества калибровки на каждом шлюзе

### 5. G-01 — Оптимизация программы сварки и НК на критическом пути

**Консенсус: 8/8.** Сварка ГЦТ — абсолютный критический путь основного периода.

**Финальная конфигурация:**

- Моделировать **семейства стыков**, не «сварку вообще»
- Учитывать нестационарность: кривая обучения бригад, усталость, сезонные факторы
- Корреляция дефектов внутри партии (плохая присадочная проволока)
- Коэффициент «строительной доступности» в целевой функции
- Автоматизированный сбор данных с оборудования для минимизации искажений
- Интеграция с графиком передачи помещений (M-03)

### 6. C-01 — GNN для управления изменениями (гетерогенный граф)

**Консенсус: 7/8.** Неконтролируемые изменения — главный драйвер перерасхода.

**Финальная конфигурация:**

- Поэтапное внедрение: старт с подграфа реакторного здания (системы 1-го и 2-го класса безопасности)
- Сначала упрощённый графовый движок на правилах, GNN — для сложных каскадов
- NLP для структурирования текстовых change orders перед подачей в GNN
- Обязательная трассировка влияния для управленческих решений
- Регламент актуализации BIM-модели как часть мастер-графика

### 7. J-01 — Вероятностная модель регуляторных инспекций и hold-points

**Консенсус: 7/8.** Инспекции — скрытый критический путь, не формализованный в стандартном CPM.

**Финальная конфигурация:**

- Позиционировать как оценку «вероятности сдачи с первого раза», не «предсказание регулятора»
- Сценарные режимы: строгий / базовый / либеральный регулятор
- Триггеры внешних событий (аварии на других АЭС, выборы, смена руководства)
- Самообучение: автоматическое обновление параметров после каждой инспекции
- Наихудшие сценарии по Вальду для планирования резервов

### 8. L-01 — Оптимизация offset между двумя блоками

**Большинство: 6/8.** Стратегическое решение для серийного строительства.

**Финальная конфигурация:**

- Использовать для **анализа чувствительности** заданного offset, а не его «назначения»
- Оптимизировать внутри допустимого диапазона (с учётом финансирования, лицензий, МПС)
- Штрафные функции за «ресурсный каннибализм» при отставании первого блока
- Выдавать **диапазон с буфером**, а не детерминированную точку
- Отдельные сценарии по learning curve (консервативный/базовый/оптимистичный)

### 9. A-02 — Адаптивный контур управления поставками ОДЦИ

**Большинство: 6/8.** Поставки ОДЦИ — один из самых жёстких ограничителей графика.

**Финальная конфигурация:**

- Замкнутый цикл: PERT → байесовское обновление → ML-триггер → локальная оптимизация → пересчёт графика
- Уровни доверия к источникам данных (ERP, FAT, инспекции, фотофиксация, логистика, таможня)
- Три класса риска по поставщикам
- Для критичных позиций (корпус реактора, ГЦНА, ПГ) — обязательная независимая верификация
- Учёт сезонности, санкций и логистических рисков для экспортных проектов

### 10. A-06 — Полная модель ПНР как отдельный «мини-проект»

**Большинство: 6/8.** Закрывает финальную и самую непредсказуемую фазу строительства.

**Финальная конфигурация:**

- Два режима: **стратегический** (полная GERT-модель для планирования) и **оперативный** (упрощённая для текущего подэтапа)
- Пилот на 1–2 критичных системах, затем расширение
- Включает сложнейшие петли: возврат при дефектах на ХГО, неуспешный физпуск → повтор
- Критерий Вальда для гарантированного пуска
- Укрупнённые семейства событий (не избыточная атомизация)

## СОГЛАСОВАННЫЕ РЕШЕНИЯ ПО СПОРНЫМ ВОПРОСАМ

### A-05 (Цифровой двойник): исключена из ТОП-10, сохранена как архитектура

**Решение единогласно (8/8):** A-05 — не отдельная внедряемая модель, а **архитектурная платформа**, в которую встраиваются все 10 идей. В итоговом документе описывается как целевая трёхслойная архитектура:

- Слой 1 — CPM (детерминированный)
- Слой 2 — PERT/GERT/Монте-Карло (стохастический)
- Слой 3 — ML-адаптивный (GNN + anomaly detection + байесовское обновление)
- Интеграция: PIM/BIM → АСУ M2 → ТСМ NC

### B-01 (Контрактная оптимизация): не включена, в резерве

**Решение (7/8):** Юридические и политические ограничения не позволяют «адаптировать» контракт по фазам. Используется как инструмент стресс-тестирования 2–3 допустимых контрактных схем на предконтрактной стадии. Кандидат для «второй волны».

### F-02 (Краны): не включена, интегрирована в A-03

**Решение (7/8):** Конфликты за краны — частный случай ресурсного ограничения. «Зашивается» как ключевое ресурсное ограничение внутрь A-03 и G-01.

### B-02 (Инвестиционная стратегия): не включена, рекомендована как приоритет второй волны

**Решение (7/8, против — уч. 6):** Идея важна для решения о FID, но узкоспециализирована. Может быть реализована как частный случай применения A-01 (подставляем разные инвестиционные сценарии).

### M-02 (Punch list): не включена, рекомендована как подмодуль

**Решение (7/8):** Включается как подмодуль A-03 (тактический уровень) или A-06 (переход к ПНР).

## ДОРОЖНАЯ КАРТА ВНЕДРЕНИЯ (согласована 7/8 участников)

### Фаза 1 — Пилот (6–12 мес.)

**D-01** (калибровка) → **A-01** (макромодель, облегчённая версия) → **A-04** (Readiness Index)

Максимальный ROI на подготовительном этапе одного блока. D-01 запускается первой, параллельно с A-01, чтобы обеспечить достоверные параметры с самого начала.

### Фаза 2 — Расширение (12–24 мес.)

**A-02** (поставки ОДЦИ) + **G-01** (сварка и НК) + **A-03** (предиктивный критический путь, гибридная версия)

Переход к оперативному управлению основного периода. К этому моменту накоплены данные для начала обучения ML-моделей.

### Фаза 3 — Полный контур (24–36 мес.)

**C-01** (управление изменениями) + **J-01** (инспекции) + **A-06** (ПНР) + **L-01** (offset)

Замыкание контура на весь жизненный цикл. GNN получает достаточную обучающую выборку. Цифровой двойник (A-05) начинает формироваться как интеграционная платформа.

## ЗАКЛЮЧЕНИЕ

Пять раундов мозгового штурма (генерация → развитие → новые идеи → голосование → критика) и финальный синтез Раунда 6 дали устойчивый, хорошо проверенный результат. Из ~150 первоначальных предложений выделены 10 приоритетных моделей, каждая из которых прошла через критику 8 экспертов и получила конкретные корректировки для практической реализации.

Ключевые принципы финального ТОП-10:

- **Модульность:** все модели реализуются поэтапно, от MVP к полному контуру
- **Калибровка прежде всего:** D-01 обеспечивает достоверность параметров для всех остальных моделей
- **Гибридный подход к ML:** rule-based + ML, с обязательной объяснимостью
- **Сквозное покрытие:** от лицензирования до физического пуска реактора
- **Практическая реализуемость:** исключены идеи, требующие нереалистичного уровня цифровой зрелости
